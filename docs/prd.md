## 1. Goals

### 1.1 Purpose

Build a personal, self-hosted intake and retrieval system that ingests heterogeneous documents, constructs a rich knowledge representation (vector embeddings + knowledge graph), and answers natural language queries with grounded, cited responses. The system is designed for personal use at low document volume with no multi-tenancy requirements.

### 1.2 Core Capabilities

- Ingest documents of multiple formats with layout-aware parsing, adaptive chunking, image/table handling, and knowledge graph extraction
- Retrieve relevant content via hybrid search (sparse + dense) augmented by knowledge graph traversal, result fusion, and reranking
- Generate grounded natural language answers with inline citations and hallucination guardrails
- Provide on-demand thematic summarization scoped by metadata filters
- Expose all functionality through REST API, MCP server, and CLI interfaces

### 1.3 Non-Goals

- Multi-tenancy or per-organization data isolation
- Agentic looping within the retrieval system (agentic orchestration is the caller's responsibility)
- Real-time document ingestion (all ingestion is async)
- Rate limiting
- Max document size limits

## 2. Functional Design

### 2.1 Authentication & API Keys

Every request to the REST API and MCP server must be authenticated using an API key passed via the `Authorization: Bearer <key>` header. CLI commands authenticate via a local config file that stores the key.

**Key properties:**
- Each key is tied to a **name** (e.g. `esteban-cli`, `n8n-agent`) that is injected into every log line, providing a full audit trail of who performed which action
- The raw key is shown **once** at generation time and never stored — only a SHA-256 hash is persisted
- A short **prefix** (first 8 characters) is stored in plaintext for identification in listings
- Keys can be revoked (soft-revoke, not deleted) — revoked keys are rejected at auth time

**CLI key management commands:**
```
rag keys create --name <name>     # generates key, displays once, stores hash
rag keys list                     # shows name, prefix, created_at, last_used_at
rag keys revoke --name <name>     # soft-revokes the key
```

### 2.2 Ingestion Pipeline

All ingestion is **asynchronous**. Submitting a document returns a `job_id` immediately. The pipeline executes in the background. Clients poll for job status.

#### 2.2.1 Job Lifecycle

Every ingestion job tracks progress at the stage level. The pipeline stages in order are:

```
pending → parsing → profiling → chunking → validation → embedding → graph_extraction → graph_linking → completed
```

A failure at any stage transitions the job to `failed:<stage>` and halts the pipeline. No partial content from a failed job is visible to retrieval. Retrieval only surfaces chunks belonging to sources with a `completed` job.

**Job state model:**
- `pending` — queued, not yet started
- `processing:<stage>` — currently executing the named stage
- `completed` — all stages succeeded; source is fully available for retrieval and graph traversal
- `failed:<stage>` — halted at the named stage; error detail is stored with the job record
- `cancelled` — manually cancelled before completion; partial artifacts cleaned up

**Job retry behavior:**
- Default retry resumes from the failed stage
- Caller may optionally specify a target stage to retry from (any earlier stage); doing so triggers cleanup of all artifacts generated by stages after the target before re-execution begins
- Retry is only valid for jobs in `failed` state

**Job cancellation:**
- Only valid for jobs in `pending` or `processing` state
- Cancellation cleans up any partial artifacts (chunks, embeddings, graph nodes) generated so far

#### 2.2.2 Deduplication

Before entering the pipeline, an MD5 hash of the file content is computed. If a source with the same MD5 already exists in an active (non-soft-deleted) state, the job is rejected with a `409 Conflict` response identifying the existing source.

#### 2.2.3 Document Storage

Ingested files are moved to a configurable storage base path (`STORAGE_BASE_PATH` env var), organized as:

```
{STORAGE_BASE_PATH}/{source_id}/{version}/original_{filename}
```

Files are never modified after storage. The storage backend interface is abstracted so it can be swapped for S3-compatible object storage in the future without changing the rest of the system.

#### 2.2.4 Metadata Capture

After storage, base metadata is recorded:

| Field | Source |
|---|---|
| `file_name` | Original filename |
| `file_type` | Detected MIME type |
| `date_ingested` | System timestamp |
| `md5` | Computed hash |
| `source_id` | System-generated UUID |
| `version` | Integer, starts at 1 |

A **lightweight LLM call** then extracts additional metadata:
- `kind` — content type (transcript, paper, article, report, infographic, etc.)
- `author` — if detectable
- `source` — publication or origin if detectable
- `domain` — legal, financial, technical, medical, general

Callers may also pass explicit metadata at submission time (e.g. `ingest_podcast` sets `kind=podcast`). Caller-supplied values take precedence over LLM-extracted values.

#### 2.2.5 File Parsing

The parser layer performs **layout analysis**, not raw text extraction. It must:
- Detect reading order, heading hierarchy, and element relationships
- Produce a document element tree: `Document → Section → [Heading, Paragraph, Table, Figure]`
- Chunks must respect element boundaries; headings are prepended to child chunks for context

**Supported formats:** PDF, DOCX, PPTX, plain text (TXT, MD)

**Embedded images:**
- OCR is run on each image to extract embedded text, associated back to the parent section
- A natural language caption is generated describing the image content and indexed as a text chunk with a pointer back to the original image
- At retrieval time the caption is matched; the original image can optionally be surfaced for multimodal generation

**Tables:**
- Always treated as atomic units — never split mid-table
- The surrounding section heading and any table caption are prepended to the table chunk
- Each table is serialized to structured text (Markdown or key-value per row) so relational meaning is preserved in the embedding
  - Example: a cell value `42%` becomes `Q3 gross margin: 42%`

#### 2.2.6 Adaptive Chunking

After parsing, the system selects a chunking strategy per document — not a single global strategy.

**Document profiling** (lightweight LLM call on document sample):
- Structure type: well-structured, loosely-structured, or unstructured
- Heading consistency: consistent, inconsistent, or none
- Content density: uniform or variable
- Primary content type: prose, tabular, mixed, Q&A pairs, code, transcript
- Estimated average section length: short (<300 tokens), medium (300–1000), long (>1000)
- Presence of tables and code blocks
- Domain: legal, financial, technical, general, medical, policy

**Strategy selection** (deterministic, no additional LLM call):

| Profile | Strategy |
|---|---|
| Well-structured + consistent headings | Structure-aware chunking |
| Transcript or interview | Semantic chunking |
| Uniform prose, no structure | Recursive separator |
| Everything else | Semantic chunking (fallback) |
| Sections estimated >1000 tokens (any strategy) | + Hierarchical parent-child layer |
| Domain: legal, financial, medical | + Proposition (atomic fact) chunking |

**Supported base strategies:**
- **Fixed-size** — token-based with overlap; for uniform corpora like logs
- **Recursive separator** — for general prose fallback
- **Structure-aware** — chunk at heading boundaries; prepend heading hierarchy to child chunks
- **Semantic** — embed sentences incrementally; break on cosine similarity drops
- **Proposition / atomic** — decompose passages into standalone factual claims via LLM
- **Hierarchical parent-child** — store two levels; retrieve on child chunk, surface parent chunk to the LLM for broader context

Chunk overlap is used conservatively — it is not a substitute for correct boundary detection.

**Quality validation** runs after chunking on a random sample:
- Default sample: 10% of chunks
- Domain override: legal, financial, medical → 25–30%
- First ingestion of a new document type/domain → 100% validation
- Sample size is configurable via environment variable
- Checks: is the chunk self-contained, does it have dangling references, is it too short, does it split a table
- Flagged chunks are re-processed with a fallback strategy or merged/discarded

**Chunking config is stored as metadata** alongside each chunk so the system can selectively re-ingest documents by type if the strategy changes later.

#### 2.2.7 Graph Extraction

After chunking, each chunk is processed for entity and relationship extraction. Results are merged into a master knowledge graph with deduplication across all sources.

**Graph structure:**

```
(SOURCE_A) -[INCLUDES]-> (CHUNK) -[MENTIONS]-> (ENTITY) <-[MENTIONS]- (CHUNK) <-[INCLUDES]- (SOURCE_B)
```

The graph stores three layers:
- **Entity nodes** — named concepts (organizations, people, policies, products, regulations, concepts, locations)
- **Relationship edges** — typed, directional edges between entity pairs (e.g. OWNS, GOVERNS, REQUIRES, IS_PART_OF); carries metadata: type, confidence score, source chunk reference
- **Chunk nodes** — actual text units, linked to entities via MENTIONS edges and to source documents via INCLUDES edges

**Entity extraction** (per chunk, lightweight LLM call):
- Extract named entities with type and known aliases
- Minimum supported types: ORGANIZATION, PERSON, POLICY, PRODUCT, REGULATION, CONCEPT, LOCATION
- Aliases captured at extraction time to support downstream deduplication

**Relationship extraction** (per chunk, lightweight LLM call):
- Extract typed, directional relationships between entity pairs in the same chunk
- Relationship types use a constrained domain vocabulary (15–25 types) to prevent relation type drift — enforced via output schema
- Each relationship carries a confidence score; only relationships above a configurable threshold (default: 0.75) are persisted
- Relationships must be grounded in the chunk text; ungrounded edges are discarded

**Entity deduplication** (async post-ingestion job):
- Embed entity names; merge candidates above a cosine similarity threshold (default: 0.92)
- A canonical name resolver picks one surface form per entity and re-links all edges to the canonical node
- Runs as a separate pass to avoid race conditions during bulk ingestion

**Cross-document linking** (async post-ingestion job):
- Every entity gets MENTIONED_IN edges to all chunks that reference it — enables cross-source retrieval
- Entity pairs co-mentioned across multiple chunks may get a RELATED_TO edge summarizing the relationship

A source's job is marked `completed` only after both the ingestion pipeline stages **and** the downstream async graph jobs (deduplication, cross-linking) finish successfully. The job status endpoint exposes the status of downstream jobs separately.

**Consistency failure modes and mitigations:**

| Failure Mode | Mitigation |
|---|---|
| Entity fragmentation | Alias normalization + embedding dedup |
| Relation type drift | Constrained vocabulary via output schema |
| Confidence noise | Confidence threshold filter on edge persistence |
| Hallucinated relationships | Grounding check against source chunk text |

#### 2.2.8 Versioning

If a document is re-submitted and its MD5 differs from the stored version, a new version record is created linked to the original `source_id`. The previous version's chunks, embeddings, and graph nodes are soft-deleted. Only the latest active version participates in retrieval and graph traversal. A version history endpoint returns the full version list for any source.

### 2.3 Retrieval Pipeline

The retrieval system receives a query and returns structured retrieval results only. It does not generate an answer. Agentic looping is explicitly out of scope. The caller is responsible for any downstream answer generation.

**Response shape:**
```json
{
  "retrieval_results": [
    {
      "score": 0.0,
      "chunk": "",
      "chunk_id": "",
      "source_id": "",
      "source_path": "",
      "source_metadata": {},
      "related": [
        {
          "entity": "",
          "chunks": [
            {
              "score": 0.0,
              "chunk": "",
              "chunk_id": "",
              "source_id": "",
              "source_path": "",
              "source_metadata": {}
            }
          ],
          "second_level_related": [
            {
              "entity": "",
              "relationship": {
                "label": "",
                "metadata": {}
              },
              "chunks": [
                {
                  "score": 0.0,
                  "chunk": "",
                  "chunk_id": "",
                  "source_id": "",
                  "source_path": "",
                  "source_metadata": {}
                }
              ]
            }
          ]
        }
      ]
    }
  ]
}
```

Root results are sorted by relevance score descending.

**Query parameters:**
- `query` — natural language question (required)
- `source_filter` — optional list of source IDs or metadata filters to scope retrieval
- `seed_count` — number of top reranked root chunks to expand into graph retrieval (default: 10, caller-configurable)
- `result_count` — number of final root retrieval results to return after expansion and final reranking (default: 5, caller-configurable)
- `rrf_k` — reciprocal rank fusion constant (default: 60, caller-configurable)
- `entity_confidence_threshold` — minimum relationship confidence for graph traversal edges (default: 0.75, caller-configurable)
- `first_hop_similarity_threshold` — minimum similarity for first-hop entity chunk selection (default: 0.5, caller-configurable)
- `second_hop_similarity_threshold` — minimum similarity for second-hop entity chunk selection (default: 0.5, caller-configurable)

**Environment-variable defaults:**
- `RETRIEVAL_RRF_K=60`
- `RETRIEVAL_RRF_SCORE_FLOOR=0.0`
- `RETRIEVAL_SEED_COUNT=10`
- `RETRIEVAL_RESULT_COUNT=5`
- `RETRIEVAL_MAX_DECOMPOSED_QUERIES=5`
- `RETRIEVAL_FIRST_STAGE_TOP_N=20`
- `RETRIEVAL_FUSED_CANDIDATE_COUNT=50`
- `RETRIEVAL_ENTITY_SELECTION_COUNT=5`
- `RETRIEVAL_SECOND_HOP_SELECTION_COUNT=5`
- `RETRIEVAL_FIRST_HOP_CHUNK_COUNT=5`
- `RETRIEVAL_SECOND_HOP_CHUNK_COUNT=5`
- `RETRIEVAL_FIRST_HOP_SIMILARITY_THRESHOLD=0.5`
- `RETRIEVAL_SECOND_HOP_SIMILARITY_THRESHOLD=0.5`
- `RETRIEVAL_ENTITY_CONFIDENCE_THRESHOLD=0.75`
- `RETRIEVAL_MAX_GRAPH_LLM_CALLS=100`
- `RETRIEVAL_MAX_GRAPH_EXPANSION_MS=4000`

#### 2.3.1 Query Variant Generation

Before retrieval, the raw query is transformed into a bounded set of query variants. This stage uses `google/gemini-2.5-flash-lite` and must produce structured output.

Techniques applied:
- **Original query** — unchanged user query
- **HyDE** — generate a hypothetical answer passage used for dense retrieval only
- **Query expansion** — add synonyms, aliases, abbreviations, and related terms
- **Step-back prompting** — rewrite the query into a more general background-oriented form
- **Decomposition** — break the query into up to `RETRIEVAL_MAX_DECOMPOSED_QUERIES` independent retrieval sub-queries

**Important constraint:** with the default configuration, the total query count is up to 9, not 10:
- 1 original
- 1 HyDE
- 1 expanded
- 1 step-back
- up to 5 decomposed

The query-variant generator must return JSON with this shape:

```json
{
  "original": "",
  "hyde": "",
  "expanded": "",
  "step_back": "",
  "decomposed": ["", ""]
}
```

The generator must avoid near-duplicate variants. Redundant variants may be dropped before retrieval.

#### 2.3.2 First-Stage Hybrid Retrieval

All query variants are executed in parallel.

For each variant:
- Run **vector search** for all variants
- Run **BM25 sparse retrieval** for all variants except the HyDE variant
- Retrieve the top `RETRIEVAL_FIRST_STAGE_TOP_N` chunks per retriever per variant
- Apply source filters before retrieval whenever possible

This produces a set of ranked candidate lists. Dense and sparse results are deduplicated by `chunk_id` within each list before fusion.

#### 2.3.3 First-Stage Fusion

All first-stage candidate lists are merged with **weighted Reciprocal Rank Fusion (RRF)**.

Default fusion weights:
- Original query: `1.00`
- Decomposed queries: `1.00`
- Expanded query: `0.85`
- Step-back query: `0.75`
- HyDE dense query: `0.65`

RRF is rank-based and is not a calibrated similarity score. The implementation must not treat the fused score as a semantic confidence score. If a score floor is used, it is a coarse pruning mechanism only.

After fusion:
1. Deduplicate by `chunk_id`
2. Keep the top `RETRIEVAL_FUSED_CANDIDATE_COUNT` chunks
3. Pass the fused set to first-stage reranking

#### 2.3.4 First-Stage Reranking and Seed Selection

The fused candidate set is reranked against the **original user query only** using `cohere/rerank-v4.0-fast`.

Requirements:
- Reranking is mandatory
- The reranker operates on `(original_query, chunk)` pairs
- The reranker score is the primary signal for seed selection
- The system keeps the top `RETRIEVAL_SEED_COUNT` chunks as root retrieval seeds

These root retrieval seeds are the only entry points for graph expansion. Graph traversal does not run independently from query entities alone.

#### 2.3.5 Seeded Graph Expansion

Graph expansion runs after first-stage reranking and starts from each root seed chunk.

For each root seed chunk:
1. Load the linked entities from the graph
2. Use `google/gemini-2.5-flash-lite` to select the top `RETRIEVAL_ENTITY_SELECTION_COUNT` entities worth exploring, considering:
   - original user query
   - root seed chunk
   - entity name
   - entity type
   - mention metadata, if available

For each selected entity, run two paths in parallel.

**Path A — first-hop entity chunk expansion**
1. Use `google/gemini-2.5-flash-lite` to formulate a query variation using:
   - original user query
   - root seed chunk
   - selected entity
2. Score that query variation against all chunks linked to the selected entity
3. Keep only chunks with similarity ≥ `RETRIEVAL_FIRST_HOP_SIMILARITY_THRESHOLD`
4. Return at most `RETRIEVAL_FIRST_HOP_CHUNK_COUNT` chunks

**Path B — second-hop entity expansion**
1. Load the entities connected to the selected entity, along with relationship label and metadata
2. Use `google/gemini-2.5-flash-lite` to evaluate which second-hop entities are worth exploring, considering:
   - original user query
   - root seed chunk
   - selected entity
   - relationship label
   - relationship metadata
   - candidate second-hop entity
3. Keep the top `RETRIEVAL_SECOND_HOP_SELECTION_COUNT` second-hop entities
4. For each selected second-hop entity, use `google/gemini-2.5-flash-lite` to formulate a query variation using:
   - original user query
   - root seed chunk
   - selected entity
   - relationship label
   - relationship metadata
   - second-hop entity
5. Score that query variation against all chunks linked to the second-hop entity
6. Keep only chunks with similarity ≥ `RETRIEVAL_SECOND_HOP_SIMILARITY_THRESHOLD`
7. Return at most `RETRIEVAL_SECOND_HOP_CHUNK_COUNT` chunks per second-hop entity

#### 2.3.6 Expansion Budgets

Graph expansion must stay bounded. The implementation must enforce hard limits.

Required controls:
- Maximum root seed chunks = `RETRIEVAL_SEED_COUNT`
- Maximum first-hop entities per seed = `RETRIEVAL_ENTITY_SELECTION_COUNT`
- Maximum second-hop entities per first-hop entity = `RETRIEVAL_SECOND_HOP_SELECTION_COUNT`
- Maximum graph-stage LLM calls = `RETRIEVAL_MAX_GRAPH_LLM_CALLS`
- Maximum graph expansion wall time = `RETRIEVAL_MAX_GRAPH_EXPANSION_MS`

When a budget is reached, the system must stop lower-priority branches first and return the best partial retrieval result available.

#### 2.3.7 Retrieval Result Construction

The system constructs one root retrieval result per seed chunk.

Each root retrieval result contains:
- the root chunk and its retrieval metadata
- the selected first-hop entities
- the Path A chunks for each first-hop entity
- the Path B second-hop entities, relationship metadata, and retrieved chunks

Within each root result:
- Deduplicate chunks by `chunk_id`
- Preserve relationship metadata for second-hop results
- Preserve path provenance for debugging and observability

A chunk may appear in more than one root result, but it may appear only once inside a given root result branch.

#### 2.3.8 Final Deduplication and Reranking

After graph expansion, the system performs final reranking at the **root result** level against the original user query.

Preferred implementation:
1. Rerank the root chunk against the original query with `cohere/rerank-v4.0-fast`
2. Rerank all related chunks against the original query with `cohere/rerank-v4.0-fast`
3. Aggregate those scores into a final root-result score

Default aggregation:
```text
final_score =
  0.60 * root_chunk_rerank_score +
  0.25 * max(first_hop_chunk_rerank_scores) +
  0.15 * max(second_hop_chunk_rerank_scores) +
  multi_path_bonus
```

`multi_path_bonus` is a small additive boost for evidence surfaced through multiple independent retrieval paths.

After final reranking:
- Deduplicate root results by `chunk_id`
- Keep the top `RETRIEVAL_RESULT_COUNT` root results
- Return only `retrieval_results`
- Do not generate an answer inside the retrieval system




### 2.4 Community Summarization

A standalone capability — separate from the per-query retrieval pipeline — that detects thematic communities across a scoped set of sources using graph clustering and optionally summarizes each community with an LLM.

Community summaries are generated **on demand** against a scoped source set — they are not pre-computed at ingestion time.

#### 2.4.1 Scope Resolution

The caller defines the working set of sources using one of three modes:

**`ids` mode** — explicit list of source IDs. No further resolution needed; the provided IDs are the scope.

**`search` mode** — one or more free-text criteria strings. Each criterion runs a `hybrid_search` operation (same dense + sparse + RRF pipeline as `rag search`). The unique `source_id` values from all results across all criteria are unioned into the working set. Inherits `--limit` and `--min-score` from `rag search` as optional overrides.

**`retrieve` mode** — one or more free-text criteria strings. Each criterion runs the full `retrieve()` pipeline — query variant generation, first-stage hybrid retrieval, graph expansion, and reranking. The unique `source_id` values from all `retrieval_results` (including source IDs from related chunks) are unioned into the working set. Inherits all optional parameters from `rag retrieve`: `--seed-count`, `--result-count`, `--rrf-k`, `--entity-confidence-threshold`, `--first-hop-similarity-threshold`, `--second-hop-similarity-threshold`, and `--trace`.

`search` and `retrieve` modes accept `--filter key=value` (repeatable, AND semantics) to apply metadata constraints to chunk retrieval during scope resolution. Multiple criteria strings run independently; their source ID results are unioned.

Sources in the working set that have zero entity connections in the graph are excluded from clustering and recorded in `metadata.sources_excluded` in the output.

#### 2.4.2 Graph Construction

The entity co-occurrence graph is built entirely **in memory at runtime** — nothing is written to Postgres or Memgraph. The graph is discarded after the clustering run.

**Step 1 — Load chunk→entity edges**

Query Memgraph for all `(Chunk)-[:MENTIONS]->(Entity)` edges where `chunk.source_id` is in the working set. This produces the base entity-chunk incidence map.

**Step 2 — Chunk-level co-occurrence edges**

For each chunk, every pair of entities that co-appear in that chunk receives an edge. Edge weight formula:

```
base_weight     = shared_chunk_count / sqrt(degree_A * degree_B)
cross_source    = 1 if entity_A and entity_B come from different sources, else 0
edge_weight     = base_weight * (1 + 0.5 * cross_source)
```

Where `degree` is the number of distinct chunks an entity appears in. The cross-source multiplier rewards entity pairs that co-occur across document boundaries, increasing the probability that Leiden detects cross-source communities.

**Step 3 — Source-level co-occurrence edges (option 3)**

For each source in the working set, every entity pair that shares the source but no common chunk receives a weak synthetic edge with weight `COMMUNITY_SOURCE_COOC_WEIGHT` (default: `0.1`). This edge is only added if no chunk-level edge already exists for the pair. This broadens the graph for sources with sparse chunk overlap.

**Step 4 — Semantic cross-source edges (option 2)**

For every pair of entities that come from **different sources**, compute the cosine similarity of their stored embeddings (loaded from the Postgres `entities.embedding` column). If the similarity meets or exceeds `COMMUNITY_SEMANTIC_THRESHOLD` (default: `0.85`), a semantic edge is added with weight `cosine_similarity * 0.5`. If a weaker edge already exists for the pair (from step 3), it is upgraded to the semantic weight. Semantic edges are only considered between cross-source entity pairs.

This step allows entities that were extracted independently from different documents but represent the same concept (e.g. `Program Management Office (PMO)` and `Project Management Office`) to be connected even if they never co-appeared in a chunk.

**Step 5 — Leiden clustering**

Run the Leiden algorithm (`leidenalg.ModularityVertexPartition`) on the weighted in-memory graph. Leiden is preferred over Louvain because it produces better-connected, more reliable communities.

Communities with fewer than `COMMUNITY_MIN_COMMUNITY_SIZE` entities (default: `3`) are dropped before output.

#### 2.4.3 Chunk Scoring and Selection

For each community, every chunk that mentions at least one community entity receives a representativeness score:

```
score = entity_overlap_count² / community_entity_count
```

Where `entity_overlap_count` is the number of community member entities the chunk mentions. Squaring the overlap count rewards chunks that are densely on-topic over chunks that are broadly tangential through a single hub entity.

Chunks with `score < COMMUNITY_CUTOFF` (default: `0.5`) are excluded. If all chunks for a community fall below the cutoff, that community is omitted from the output entirely.

From the qualifying chunks, up to `COMMUNITY_TOP_K_CHUNKS` (default: `5`) are selected. For cross-source communities, selection applies source diversity pressure: chunks are interleaved by source (highest-scoring chunk from source A, then source B, then back to A) to ensure multi-source representation in the output.

#### 2.4.4 Optional Summarization

When `--summarize <model_name>` is provided, each community's selected chunks are passed to the specified OpenRouter model with the following prompt:

```
Craft a compelling narrative summarizing these chunks of related information

<chunks>
{chunk_1_content}

---

{chunk_2_content}
...
</chunks>
```

The prompt template is overridable via the `COMMUNITY_SUMMARIZATION_PROMPT` environment variable. The `summary` field in the output is populated with the LLM response. When `--summarize` is not provided, `summary` is an empty string.

Summarization calls are made per community and are not parallelized by default.

#### 2.4.5 Output Schema

Communities are ordered: cross-source communities first, then by entity count descending within each group.

```json
{
  "metadata": {
    "scope_mode": "ids | search | retrieve",
    "source_count": 4,
    "sources_excluded": [
      { "source_id": "", "reason": "no entity connections" }
    ],
    "parameters": {
      "semantic_threshold": 0.85,
      "source_cooc_weight": 0.1,
      "cutoff": 0.5,
      "min_community_size": 3,
      "top_k_chunks": 5
    }
  },
  "communities": [
    {
      "community_id": "",
      "is_cross_source": true,
      "entity_count": 0,

      ],

          "chunk_id": "",

        }

    }
  ]
}
```

## 3. System Design

### 3.1 Infrastructure

**Deployment:** Docker Compose (single-host, self-hosted)

**Data stores:**

| Layer | Technology | Responsibility |
|---|---|---|
| Relational + Vector | PostgreSQL + pgvector + pg_bm25 (ParadeDB) | Source records, chunk metadata, embeddings, job queue, API keys, audit log, BM25 index |
| Graph | Memgraph | Entity nodes, relationship edges, chunk nodes (topology only — data lives in Postgres) |

**Key architectural point:** Memgraph stores graph topology only. `chunk_id` and `entity_id` values in Memgraph are foreign keys back to Postgres rows. Graph traversal returns IDs → chunk text is hydrated from Postgres in a single batch query.

**Job queue:** pg-based queue (e.g. `pgqueuer`) — sufficient for personal-use ingestion volumes (dozens of documents per day). No external broker needed.

**Observability stack (included in Docker Compose):**
- Loki + Grafana — log aggregation and dashboards
- Prometheus + Grafana — metrics (reuses same Grafana instance)
- OpenTelemetry → Tempo — distributed tracing across pipeline stages

**Health endpoints:** `/health/live` and `/health/ready` (checks Postgres + Memgraph connectivity)

### 3.3 Graph Model (Memgraph)

Nodes:
- `(:Source {source_id})` — references Postgres sources
- `(:Chunk {chunk_id})` — references Postgres chunks
- `(:Entity {entity_id, canonical_name, entity_type})`

Edges:
- `(Source)-[:INCLUDES]->(Chunk)`
- `(Chunk)-[:MENTIONS {confidence}]->(Entity)`
- `(Entity)-[:RELATED_TO {type, confidence, chunk_id}]->(Entity)`
- `(Entity)-[:MENTIONED_IN]->(Chunk)` (cross-document)

### 3.4 Model Configuration

**No model names are hardcoded in the application.** All models are configured via environment variables with defaults. Models are accessed via OpenRouter.

| Pipeline Role | Env Var | Recommended Model | Rationale |
|---|---|---|---|
| Metadata extraction | `MODEL_METADATA_EXTRACTION` | `google/gemma-3-4b-it` | Simple classification/extraction; smallest capable model |
| Document profiling | `MODEL_DOC_PROFILING` | `google/gemma-3-4b-it` | Structured output from document sample; low complexity |
| Chunk quality validation | `MODEL_CHUNK_VALIDATION` | `qwen/qwen2.5-7b-instruct` | Requires reading comprehension; slightly more capable |
| Entity extraction | `MODEL_ENTITY_EXTRACTION` | `qwen/qwen2.5-7b-instruct` | Named entity recognition with typed output schema |
| Relationship extraction | `MODEL_RELATIONSHIP_EXTRACTION` | `qwen/qwen2.5-7b-instruct` | Constrained vocabulary output; 7B sufficient |
| Proposition chunking | `MODEL_PROPOSITION_CHUNKING` | `qwen/qwen2.5-14b-instruct` | Atomic fact decomposition requires stronger reasoning |
| Query rewriting / HyDE | `MODEL_QUERY_REWRITING` | `qwen/qwen2.5-14b-instruct` | Multi-technique rewriting; benefits from larger context understanding |
| Graph traversal scoring | `MODEL_GRAPH_TRAVERSAL_SCORING` | `qwen/qwen2.5-7b-instruct` | Binary relevance judgment per chunk; lightweight |
| Reranking | `MODEL_RERANKER` | `qwen/qwen2.5-14b-instruct` | Cross-encoder scoring; needs solid reasoning but runs on short (query, chunk) pairs |
| Answer generation | `MODEL_GENERATION` | `qwen/qwen2.5-72b-instruct` | Final answer quality directly impacts user experience; use the largest affordable model |

All OpenRouter API calls share a single `OPENROUTER_API_KEY` environment variable. Timeouts, max retries, and temperature defaults per role are also configurable via environment variables.

### 3.5 Error Handling & Observability

**Structured logging:**
Every log line is JSON (via `structlog`) and includes at minimum:

```json
{
  "timestamp": "",
  "level": "",
  "job_id": "",
  "api_key_name": "",
  "stage": "",
  "action": "",
  "duration_ms": 0,
  "model_used": "",
  "token_count": 0,
  "status": "",
  "error": null
}
```

`api_key_name` is resolved at auth time and injected into the logging context for the duration of the request — all log lines from that request carry it automatically.

**Per-stage failure handling:**
- Each pipeline stage writes its result and any error to the `stage_log` JSONB column on the job record before exiting
- Failed stages store the full error, traceback, and the partial output (if any) for debugging
- Failed jobs remain in the database indefinitely (not auto-cleaned) and are surfaced via `rag jobs list --status failed`

**Retry with stage rollback:**
When retrying from a stage earlier than the failure point, the system deletes all artifacts produced by stages after the target stage before re-executing:
- Stages that produce Postgres rows (chunks, entities): rows are hard-deleted for the job
- Stages that produce Memgraph nodes: nodes referencing the job's chunks are deleted
- Stages that produce embeddings: pgvector rows are deleted

**Async job failure handling:**
Downstream async jobs (entity dedup, cross-document linking) have their own status tracked in the job record under `stage_log`. If they fail, the source remains in a `pending_graph` state — not surfaced for retrieval — and the failure is visible in job status.

**Observability metrics (Prometheus):**
- Ingestion job count by status and stage
- Stage duration histograms per stage and model
- LLM call count, token usage, and latency per model and role
- Retrieval query latency (end-to-end and per sub-stage)
- Graph traversal candidate count and pruning rate
- BM25 and vector result counts pre/post fusion

**Health checks:**
- `GET /health/live` — process is alive
- `GET /health/ready` — Postgres and Memgraph connections are healthy
