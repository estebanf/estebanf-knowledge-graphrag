# Image Description Design

**Date:** 2026-04-24
**Status:** Approved

## Problem

When docling converts PDF, DOCX, and PPTX files to markdown, images are emitted as `<!-- image -->` placeholders. Markdown files ingested with local image references (`![alt](relative/path)`) pass those references through as inert text. In both cases, the image content is invisible to downstream chunking, embedding, and graph extraction — 126 sources and 1,817 chunks are currently affected.

## Goal

Replace every image placeholder or inline image reference with a concise narrative description generated by a multimodal LLM, inline during the parsing stage, before the markdown is stored or processed further.

## Scope

- PDF, DOCX, PPTX: replace `<!-- image -->` placeholders produced by docling
- Markdown: replace `![alt](local_path)` references where the image file exists on disk
- Markdown with remote URLs (`https://`) or data URIs: out of scope, left as-is
- Remediation of the 126 existing affected sources

## Architecture

### Where image description happens

Image description runs **inline in the parsing stage**, after the document is converted but before `ParseResult` is returned. This means the `markdown_content` stored in `sources` already contains descriptions, and no other stage needs to change.

### New module: `src/rag/image_description.py`

Single public function:

```python
def describe_image(image_bytes: bytes, mime_type: str) -> str
```

Encodes the image as base64 and calls OpenRouter's chat completions API using `settings.MODEL_IMAGE_DESCRIPTION`. The prompt asks for a concise, factual description suitable for a knowledge base. Raises `httpx.HTTPStatusError` on API failure, which the caller wraps into `ParseError`.

### Changes to `src/rag/parser.py`

**PDF / DOCX / PPTX path:**

Enable `generate_picture_images=True` in `PdfPipelineOptions`. After docling converts the document, iterate `result.document.pictures` in document order — each item exposes a PIL image, which is encoded to PNG bytes. For each picture, call `describe_image` and replace the next `<!-- image -->` occurrence in the exported markdown using `str.replace('<!-- image -->', description, 1)`. Document order matches placeholder order by docling's design.

**Markdown path:**

After reading the file text, scan for `![alt](local_path)` patterns using a regex that excludes `http://`, `https://`, and `data:` prefixes. Resolve each path relative to the stored markdown file's directory. For each resolved path that exists, load the bytes, detect the MIME type from the extension, call `describe_image`, and replace the entire `![alt](path)` span with the description text. Missing image files are skipped silently.

**Error handling:**

Any exception from `describe_image` propagates out of `parse_document` and is caught by the pipeline's existing `_fail_stage(conn, job_id, "parsing", exc)` path. The job lands in `failed:parsing` and can be retried normally. There is no partial-success fallback — if one image fails, the whole parse fails.

### Changes to `src/rag/storage.py`

New function:

```python
def store_markdown_images(source_id: str, markdown_path: Path, version: int) -> None
```

Called from `submit_ingestion_job` immediately after `store_file` when the file type is `md` or `markdown`. Scans the markdown content for local image references, resolves them relative to `markdown_path.parent`, and copies each existing image into the versioned storage directory preserving the relative path. Missing images are logged and skipped — the parser will also skip them at parse time.

This step is necessary because the async worker processes the file from versioned storage, not from its original location, so image files must be present in storage before the job runs.

### Changes to `src/rag/config.py`

```python
MODEL_IMAGE_DESCRIPTION: str = "google/gemini-2.0-flash-lite"
```

`gemini-2.0-flash-lite` is fast, cheap, and reliably multimodal. The model role follows the existing `MODEL_*` naming convention and can be overridden via `.env`.

## Remediation

### Current state

| File type | Affected sources | Affected chunks |
|-----------|-----------------|-----------------|
| PDF       | 123             | 1,770           |
| PPTX      | 3               | 47              |
| **Total** | **126**         | **1,817**       |

### Approach

Remediation follows the existing pattern in `src/rag/remediation.py`. The image remediation restarts from `parsing` (not `profiling`) because the fix changes the markdown itself — `sources.markdown_content` must be regenerated with descriptions in place of placeholders.

`cleanup_from_stage("parsing")` deletes chunks and graph artifacts for the source. The job is then reset to `pending` with `retry_from_stage = 'parsing'`, and the worker re-runs the full pipeline.

### New additions to `src/rag/remediation.py`

- SQL query that identifies completed jobs with at least one chunk containing `<!-- image -->`
- `AffectedImageSource` dataclass (mirrors `AffectedSource`)
- `remediate_image_sources(dry_run, only_source_id, limit)` — same interface as `remediate()`
- `remediate_image_source(conn, driver, source_id, job_id, file_name)` — cleans up from `parsing`, resets job to pending

### New CLI command

```
rag remediate images [--dry-run] [--source-id UUID] [--limit N]
```

Same flags as the existing `rag remediate` command. `--dry-run` prints affected sources without modifying anything. After running, start the worker to process the re-queued jobs.

## What does not change

- `STAGE_ORDER` and pipeline orchestration
- Chunking, embedding, graph extraction, graph linking
- The `ParseResult` datatype
- Any database schema
